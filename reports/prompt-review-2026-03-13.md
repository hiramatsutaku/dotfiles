## 4. プロンプティング力の評価

### 強み

- **具体的な参照リソースの提示**: 「https://github.com/xxxxxxx/applications/blob/main/.github/workflows/build.yml を参考にpushするようにして」「rq-applicationsリポジトリと同様に、lefthook, oxlint, oxfmt, tsgoでお願い」のように、既存の実装を参照先として明示するパターンが頻出。AIに対して十分なコンテキストを与えている
- **GitHub URL直接提示によるコンテキスト共有**: CI失敗時に「https://github.com/.../actions/runs/... がダメそう」とURLを直接貼り、エラーログも併記。AIが状況を正確に把握できるようにしている
- **アーキテクチャ方針の明確な提示**: 「一度リソースを全部消してやり直していいと思ってるんだよね。rdsを他と共有するつもりだし。」のように、技術的な意思決定の背景と方針を伝えた上で作業を依頼している
- **批判的レビュー依頼**: 「厳しめにレビューして」「この変更は意味不明」「s3://dev/dev/ってなんか微妙じゃない？」と、AIの出力に対して能動的にフィードバックを返している

### 改善ポイント

- **曖昧な短文指示の頻出**: 「pr」「push」「fix」「続けて」のような1-2語の指示が多く、AIに暗黙のコンテキスト共有を前提としている。セッション途中では問題ないが、新セッション開始時に同じスタイルだと誤解が生じるリスクがある → **改善提案**: 新セッション開始時は「[リポジトリ名]の[ブランチ名]でPR作成して」のように最低限のコンテキストを付与する
- **要件の後出し**: 「別PRにしてー　PRマージされてたら新たい作るようにしてよ。何回も言ってるでしょ」と、本来最初に指定すべきPR運用ルールが後から出ている。また「mainじゃなくてdevelop！」のように基本設定の補足が後出しになるケースがある → **改善提案**: プロジェクトごとの基本ルール（ベースブランチ、PR運用方針等）をCLAUDE.mdに記載し、毎回の指示で繰り返す必要をなくす
- **誤字・途中送信が多い**: 「deploymentsん方」「話にブルーリクエストにして」「formst」など、誤字や音声入力由来と思われる不完全な入力が散見される。AIは推測で補完するが、意図が正しく伝わらないリスクがある → **改善提案**: 重要な指示は送信前に一度確認する。特に音声入力使用時は注意

### 特徴的な癖

- 短い肯定応答として「hai」（ローマ字の「はい」）を頻用している（20回以上出現）
- 「pr」の一言でPR作成を指示するパターンが非常に多い（15回以上）。Claude Codeとの暗黙の了解が確立されている
- エラー発生時にGitHub Actions URLをそのまま貼り付ける習慣があり、効率的なデバッグフローが定着している
- 「/coderabbit-fix」スラッシュコマンドを積極的に活用している（10回以上）
- 日本語と英語が自然に混在しており、技術用語は英語のまま使用するスタイル

## 5. AI活用スタイル

### 主体的に活用できている領域

- **インフラ設計の意思決定**: 「vpc-lattice, ecs, security-groups, dnsはまとめていいかもねー」「うん。でも良く考えたら、aws/modules/platform/keycloak/vpcじゃなくてaws/modules/platform/vpcこうあるべきだと思うんだよね。」のように、モジュール構造やリソース配置の方針はユーザーが決定し、AIに実装を委任するパターンが確立されている
- **コードレビューとフィードバック**: AIが生成したコードに対して「この変更は意味不明」「レガシーにしないで」「PRに余計な差分がある気がするよ」と的確にレビューし、修正指示を出している
- **デプロイ戦略の設計**: 「mainマージでdev環境へデプロイ。リリースノート作成で本番環境へデプロイ。」「plan確認してからにしたいね」と、CI/CDの運用方針を自ら決定している

### AIへの依存が高い領域

- **Amplifyの詳細設定**: Amplifyのプレビュー環境構築で「PRプレビュー使うんだっけ？」「始まらない。。」「手動デプロイ API は SSR アプリに対応していない...なぜ？」と、Amplifyの仕様理解にAIを頼りつつ試行錯誤が長引いている → **自立に向けた提案**: Amplifyの公式ドキュメントでSSR制約とカスタムドメイン設定を一度体系的に確認すると、試行錯誤の時間を短縮できる
- **CI/CDエラーのデバッグ**: GitHub ActionsのURLとエラーログを貼り付けて「落ちてる原因を調べて」「ダメだね」「まだダメー」と、AIにデバッグを繰り返し依頼するパターンが目立つ。特にIAM権限エラーやSSMパラメータ不足のトラブルシューティングで顕著 → **自立に向けた提案**: よくあるエラーパターン（PermissionsBoundary違反、ParameterNotFound等）のトラブルシューティングチェックリストを作成しておくと、AI往復回数を減らせる

### ツール別の活用傾向

- **Claude Code**: 唯一検出されたツール。インフラ構築からフロントエンド修正、PR作成、CIデバッグまで幅広く活用。特にマルチリポジトリ（rq-infra、rq-infra-platform、rq-deployments、rq-applications等）を横断した作業を1つのAIセッションで指示するスタイルが特徴的。`/coderabbit-fix`や`/daily-report`などスラッシュコマンドも積極利用している

## 6. 成長の軌跡と学習提案

### 観察された成長

- 期間が7日間と短いため、大きな成長トレンドの観察は難しいが、Amplifyカスタムドメイン設定において「console-preview.dev.recerqa.com」→「preview.console.dev.recerqa.com」→「*.console.dev.recerqa.com」と、ドメイン設計の理解が短期間で深化している
- 週の前半はECS/ecspressoのデプロイパイプライン構築に集中し、後半はAmplifyプレビュー環境やtechblog構築に移行しており、複数のインフラ課題を並行して処理する能力が見られる

### 繰り返し発生する課題

- **PRマージ後の新規PR作成ルールの不徹底**: 「別PRにしてー　PRマージされてたら新たい作るようにしてよ。何回も言ってるでしょ」と繰り返しフラストレーションが発生している
- **IAM権限不足によるデプロイ失敗**: PermissionsBoundary違反、sts:TagSession不許可、SSMパラメータ不足など、権限関連のエラーが複数回発生し、その都度AIと試行錯誤している
- **PR説明（description）の崩れ**: 「PR descriptionが崩れているので修正してー」「PRの説明も修正してー」と複数回指摘しており、AIのPR生成品質に課題がある

### 推奨される次のステップ

1. **CLAUDE.mdへのプロジェクトルール追記**: 「PRマージ済みなら新規PR作成」「ベースブランチはdevelop（pj-tmpの場合）」等、繰り返し指示している内容をCLAUDE.mdに記載することで、毎回の指示コストを削減できる
2. **IAM設計パターンのドキュメント化**: クロスアカウントOIDC→AssumeRole→PermissionsBoundaryのパターンが複数リポジトリで繰り返し使われている。デシジョンレコード（docs/decisions/）にIAM設計方針を記録しておくと、新リポジトリ追加時の試行錯誤が減る
3. **長期間分析の実施**: 今回は7日間の分析であり、技術的成長の軌跡や季節的な作業パターンを把握するには不十分。`/prompt-review 90` で過去90日分の分析を行うと、より包括的な評価が可能になる
